
# Code Overview

    My design for the project took a semi-modular approach, while I felt like interface implementation was not necessary in my case as methods don't really fit interface :"creation" parameters; all my methods were apart from their organised classes, with a focus on Singleton use and Inheritance.


    HolloMon_Network/

    [HolloClient.java] - This class created was handled as a singleton which managed the actual connection to the Hollomon servers; its focus was primarily establishing and creating handles for the socket, whether that be written to or received from.

    [HolloSetup.java] | This class handles the user login process, along with ways for handling arbitrary user input, along with logging in the user.



    HolloMon_Log/

    [HolloLog.java] - This class serves the purpose of debugging, but also a way for the user to see what “backend” they are dealing with, whether that is the actual Hollomon command line interface, or the client side of it all.



    HolloMon_Features/

    [HolloMenu.java] - This class handled the menu options for the user, stored the user's credits, cards, available cards and displayed art. It used some features as part of HolloMon_Network.

    [HolloCard.java] - This class handled the card implementation for Hollomon, including features such as sorting a list of those objects based upon specific queries.

    [HolloTrade.java] - This class handles all Hollomon trading options, including the optional Hollomon Auto-Trader: Auto-Buy, Auto-Sell, Sell, Buy.

# Testing Strategy

    Logging

    My testing strategy was extremely optimised by the implementation of a logging feature; it allowed me to see where my variables values changed at what point of execution but also see specific warning signs that I implemented for debugging purposes. Very useful with testing edge-cases specifically, such as truncation of integers-longs, when dealing with server parameters and user input.

    IDE Debugger

    The approach towards testing was also amplified by my IDE’s debugging feature, which is a standard step over / step into process, which was also paired with breakpoints to see where an exception was thrown at a specific point of execution, variable null values, and such.

    My tests were implemented throughout my client, for example in the case of HolloMon_Network/HolloSetup.java, I tested the edge case of arbitrary user credentials, I padded this area to implement a check for those cases where an exception was thrown and repeated this test with parsing in different primitive/non-primitive types. In which, any form of input is now dealt with.
